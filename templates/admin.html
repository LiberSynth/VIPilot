Инициализация при старте
При запуске однократно выполняется start_main_loop: инициализируется и мигрирует БД, сбрасывается счётчик активных потоков, все батчи проверяются на наличие неизвестных статусов. Из БД читается сохранённое состояние воркфлоу — если оно было поставлено на паузу до перезапуска, состояние восстанавливается. Затем запускаются два фоновых демон-потока: главный цикл (main_loop) и keepalive-пинг.

Keepalive
Отдельный поток каждые 4 минуты делает HTTP GET на /healthz своего же домена — чтобы удержать процесс живым в Replit-окружении, которое засыпает при отсутствии трафика.

Главный цикл (main_loop)
Работает в бесконечном цикле. Каждая итерация:

  1. Пауза-контроль. В начале итерации проверяется флаг паузы. Если поставлен — поток блокируется до снятия.
  2. Чтение конфигурации из БД. Берутся актуальные значения loop_interval (по умолчанию 5 с) и max_batch_threads (по умолчанию 2).
  3. Планирование батчей. Вызывается _run_planning: читается расписание и список активных таргетов. Для окна «от текущего момента до buffer_hours часов вперёд» проверяются слоты расписания за вчера/сегодня/завтра. Если батч для слота ещё не создан — создаётся. Также обрабатывается «догон»: слоты прошлого, попавшие в горизонт предыдущего запуска.
  4. Отмена устаревших батчей. Батчи в статусе ожидания, чьи слоты расписания удалены или таргет отключён, помечаются отменёнными.
  5. Диспетчеризация пайплайнов. Если активных потоков меньше лимита — из БД забираются actionable-батчи. Пайплайн определяется по статусу батча через словарь: pending / story_generating → story; story_ready / video_generating / video_pending → video; video_ready / transcoding → transcode; transcode_ready / story_posted → publish. Для publish дополнительно проверяется, наступило ли время публикации. Каждый батч захватывается атомарно и запускается в отдельном демон-потоке; по завершении цикл немедленно пробуждается.
  6. Cleanup. Если поток очистки не запущен или завершился — стартует новый поток cleanup.run.
  7. Ожидание. Цикл засыпает на loop_interval секунд, но может быть разбужен досрочно при освобождении потока батча.

Логирование HTTP-запросов
Flask-хук before_request логирует каждый входящий запрос: метод, путь, IP, заголовки и тело (при наличии).

Суть архитектуры: однопоточный планировщик с мультипоточным исполнением пайплайнов, управляемый через состояние в БД, с явным контролем параллелизма и возможностью паузы без перезапуска процесса.
