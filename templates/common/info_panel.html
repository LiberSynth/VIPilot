Архитектура комплекса ▾

Фронтенд — HTML + JavaScript
Пользовательский интерфейс построен на HTML-шаблонах Jinja2 и ванильном JavaScript без фреймворков. Каждый модуль системы имеет собственную страницу: Director, Screenwriter, Producer, Monitor, Operator, Settings. Логика каждой страницы вынесена в отдельный JS-файл; общие компоненты — в переиспользуемые Jinja2-макросы. Взаимодействие с бэкендом — через REST API по JSON.

Веб-сервер — Python (Flask)
Flask-приложение обслуживает HTML-страницы и предоставляет REST API для фронтенда. Роуты разделены на blueprints по функциональным областям: веб-интерфейс и API. Веб-сервер запускается в основном потоке; параллельно с ним стартует фоновый поток сервисного слоя.

Сервисный слой — Python
Ядро системы: планировщик, пайплайны генерации контента (сценарий → видео → транскодирование → публикация), клиенты внешних API (LLM, видеогенерация, VK, Dzen) и вспомогательные сервисы. Работает в фоновых потоках независимо от веб-сервера. Взаимодействие с веб-сервером — только через состояние в БД.

База данных — PostgreSQL
Единственный источник истины для всего комплекса. Хранит состояние батчей (машина состояний пайплайна), сгенерированные тексты, метаданные видео, логи выполнения пайплайнов и всю конфигурацию системы. Веб-сервер и сервисный слой общаются друг с другом исключительно через БД — прямых вызовов между слоями нет.

Главный цикл сервиса ▾

Инициализация при старте
При импорте main после ensure_required_software и init_app выполняется инициализация: check_upgrade, register_blueprints, recover_interrupted_batches, environment.init() (счётчик потоков и pause/running из БД), лог «Приложение запущено», затем стартует фоновый поток main_loop.

Главный цикл (main_loop)
Работает в бесконечном цикле. Каждая итерация:

  1. Пауза-контроль. В начале итерации проверяется флаг паузы. Если поставлен — поток блокируется до снятия.
  2. Чтение конфигурации из БД. Вызывается refresh_environment(), которая читает 4 параметра: deep_debugging, loop_interval (по умолчанию 15 с), max_batch_threads (по умолчанию 5) и max_model_passes (по умолчанию 5).
  3. Планирование батчей. Вызывается planning.tick(): читается расписание и список активных таргетов. Для окна «now − buffer_minutes … now + buffer_minutes» проверяются слоты за вчера/сегодня/завтра. Если planning-батч для scheduled_at ещё не был (любой статус) — создаётся; прошлые слоты старше горизонта не догоняются.
  4. Диспетчеризация пайплайнов. Если активных потоков меньше лимита — из БД забираются actionable-батчи. Пайплайн определяется парой (type, status): planning + pending → planning; story + pending → story; movie + pending / processing / processed → video; transcode + pending / processing → transcode; publish + pending и составные статусы {slug}.{method}.pending / {slug}.{method}.completed → publish. Для publish дополнительно проверяется, наступило ли время публикации. Каждый батч захватывается in-memory через claim_batch() (добавление batch_id в множество активных — защита от дублирования) и запускается в отдельном демон-потоке; по завершении цикл немедленно пробуждается.
  5. Cleanup. Если поток очистки не запущен или завершился — стартует новый поток cleanup.run.
  6. Ожидание. Цикл засыпает на loop_interval секунд, но может быть разбужен досрочно при освобождении потока батча.

Логирование HTTP-запросов
Flask-хук before_request логирует входящие запросы: метод, путь и IP. Часть GET-запросов (монитор, статика и др.) намеренно не логируется — они слишком частые и не несут значимой информации.

Суть архитектуры: однопоточный планировщик с мультипоточным исполнением пайплайнов, управляемый через состояние в БД, с явным контролем параллелизма и возможностью паузы без перезапуска процесса.

Общая логика работы пайплайна ▾

Stage-батчи — цепочка этапов
Каждый этап — отдельная запись в batches со своим type (planning, story, movie, transcode, publish), связанная с предыдущей через batch_id_source. Диспетчер главного цикла запускает батч в отдельном потоке — именно батч, а не пайплайн: один и тот же пайплайн может одновременно обрабатывать несколько батчей в разных потоках. Переход к следующему этапу — создание нового stage-батча (db_create_transcode_batches, db_create_publish_batches) после ready на предыдущем. Пайплайны не вызывают друг друга напрямую.

Маршрут stage-батча по статусам

  1. planning, pending → planning.py: назначение видео из пула. По завершении — ready.
  2. story, pending → processing → ready → story.py: генерация текстового сценария через LLM.
  3. movie, pending → processing → processed → ready → video.py: отправка сценария провайдеру (fal.ai и др.), опрос задачи, скачивание и сохранение видео.
  4. transcode, pending → processing → ready → transcode.py: перекодирование через ffmpeg. Если транскодирование отключено — батч сразу переводится в ready.
  5. publish, pending → {slug}.{method}.pending / {slug}.{method}.completed → completed или partially → publish.py: публикация в настроенные каналы (ВКонтакте, Дзен). Батч создаётся, когда scheduled_at ≤ now + loop_interval (упреждение на один тик цикла).

Захват батча и защита от дублирования
Перед запуском пайплайна диспетчер захватывает батч через claim_batch(): batch_id добавляется в in-memory множество активных батчей. Если batch_id уже там есть — захват отклоняется, батч пропускается. Это исключает ситуацию, когда два потока одновременно берут один и тот же батч. Промежуточные статусы (например, processing) выставляются при dispatch или внутри пайплайна.

Логирование выполнения
Перед запуском пайплайна в потоке диспетчер передаёт batch_id и category в run(). Пайплайн пишет через write_log_entry(batch_id, channel, …); канал — в log_entries.channel, одна строка log на батч. Статус батча — только в batches.status; монитор: один блок на батч, строки «время — канал — текст».

Обработка ошибок
Обработка ошибок централизована в runner.py: при AppException батч переводится в статус error и ошибка фиксируется в логе. При любом другом исключении батч переводится в статус fatal_error и ошибка также фиксируется в логе. Повторный запуск батчей из статусов error и fatal_error главным циклом не предусмотрен.

Суть архитектуры пайплайнов: линейная цепочка независимых обработчиков, связанных только через статус батча в БД. Каждый пайплайн идемпотентен в части захвата и знает только свой участок работы. Повторяющийся код вынесен в общий модуль чистыми функциями — DRY без классов и наследования.
