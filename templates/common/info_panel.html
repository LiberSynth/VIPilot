Инициализация при старте
При запуске однократно выполняется start_main_loop: инициализируется и мигрирует БД, сбрасывается счётчик активных потоков, все батчи проверяются на наличие неизвестных статусов. Из БД читается сохранённое состояние воркфлоу — если оно было поставлено на паузу до перезапуска, состояние восстанавливается. Все лог-записи пайплайнов в статусе «running» переводятся в «cancelled» — чтобы после рестарта в мониторе не оставалось мигающих индикаторов незавершённых задач. Дополнительно вызывается _fix_orphaned_running(): находит лог-записи, завершившихся в статусе «running», у которых батч уже находится в финальном статусе, и переводит их в «cancelled». Затем запускаются два фоновых демон-потока: главный цикл (main_loop) и keepalive-пинг.

Keepalive
Отдельный поток каждые 4 минуты делает HTTP GET на /healthz своего же домена — чтобы удержать процесс живым в Replit-окружении, которое засыпает при отсутствии трафика.

Главный цикл (main_loop)
Работает в бесконечном цикле. Каждая итерация:

  1. Пауза-контроль. В начале итерации проверяется флаг паузы. Если поставлен — поток блокируется до снятия.
  2. Чтение конфигурации из БД. Вызывается refresh_environment(), которая читает 5 параметров: deep_logging, emulation_mode, use_donor, loop_interval (по умолчанию 15 с) и max_threads (по умолчанию 5).
  3. Планирование батчей. Вызывается planning.run(): читается расписание и список активных таргетов. Для окна «от текущего момента до buffer_hours часов вперёд» проверяются слоты расписания за вчера/сегодня/завтра. Если батч для слота ещё не создан — создаётся. Также обрабатывается «догон»: слоты прошлого, попавшие в горизонт предыдущего запуска.
  4. Отмена батчей без актуального слота. Батчи в статусе ожидания, чьи слоты расписания удалены, помечаются отменёнными.
  5. Диспетчеризация пайплайнов. Если активных потоков меньше лимита — из БД забираются actionable-батчи. Пайплайн определяется по статусу батча: pending / story_generating → story; story_ready / video_generating / video_pending → video; video_ready / transcoding → transcode; transcode_ready и составные статусы {slug}.{method}.pending / {slug}.{method}.published → publish. Для publish дополнительно проверяется, наступило ли время публикации. Каждый батч захватывается атомарно и запускается в отдельном демон-потоке; по завершении цикл немедленно пробуждается.
  6. Cleanup. Если поток очистки не запущен или завершился — стартует новый поток cleanup.run.
  7. Ожидание. Цикл засыпает на loop_interval секунд, но может быть разбужен досрочно при освобождении потока батча.

Логирование HTTP-запросов
Flask-хук before_request логирует каждый входящий запрос: метод, путь, IP, заголовки и тело (при наличии).

Суть архитектуры: однопоточный планировщик с мультипоточным исполнением пайплайнов, управляемый через состояние в БД, с явным контролем параллелизма и возможностью паузы без перезапуска процесса.

Общая логика работы пайплайна ▾

Четыре пайплайна — одна цепочка
Батч проходит через четыре последовательных пайплайна. На каждом этапе диспетчер главного цикла запускает батч в отдельном потоке — именно батч, а не пайплайн: один и тот же пайплайн может одновременно обрабатывать несколько батчей в разных потоках. Переход между пайплайнами происходит через смену статуса батча в БД: диспетчер подхватывает батч на следующей итерации и передаёт его следующему пайплайну. Пайплайны не вызывают друг друга напрямую — они полностью независимы и общаются только через состояние батча в БД.

Маршрут батча по статусам

  1. pending / story_generating → story.py: генерация текстового сценария через LLM. По завершении — статус story_ready.
  2. story_ready / video_generating / video_pending → video.py: отправка сценария в API генерации видео (fal.ai), опрос статуса задачи, сохранение результата. По завершении — статус video_ready.
  3. video_ready / transcoding → transcode.py: перекодирование видео через ffmpeg — нормализация разрешения, частоты кадров и формата. Если транскодирование отключено в настройках — батч сразу переводится в transcode_ready через skip.
  4. transcode_ready / {slug}.{method}.pending / {slug}.{method}.published → publish.py: публикация в настроенные каналы (ВКонтакте, Дзен). Пайплайн последовательно проходит по активным таргетам и включенным методам из publish_method.*. Дожидается наступления scheduled_at если время ещё не пришло. По завершении всех шагов — статус published.

Захват батча и защита от дублирования
Перед запуском пайплайна диспетчер атомарно переводит батч в промежуточный статус («захватывает» его) через UPDATE с проверкой текущего статуса. Это исключает ситуацию, когда два потока одновременно берут один и тот же батч. Пайплайн начинает работу уже с «захваченным» батчем и сразу получает его данные из БД.

Проверка отмены батча
Внутри каждого пайплайна выполняется проверка: существует ли ещё слот расписания и активен ли таргет. Если нет — батч переводится в статус cancelled, в лог пишется причина, и пайплайн завершается без ошибки. Пробный и оперативный батч проверку отмены пропускают — они живут вне расписания.

Логирование выполнения
Перед запуском пайплайна в потоке диспетчер создаёт лог-запись через write_log() в start_batch_thread и передаёт log_id в пайплайн. Пайплайн наполняет лог точками выполнения через db_log_entry. По завершении лог закрывается с финальным статусом и сообщением через db_log_update. Лог-запись привязана к batch_id и отображается в мониторе.

Обработка ошибок
Обработка ошибок централизована в runner.py: при AppException батч переводится в статус error и ошибка фиксируется в логе. При любом другом исключении батч переводится в статус fatal_error и ошибка также фиксируется в логе. Повторный запуск батчей из статусов error и fatal_error главным циклом не предусмотрен.

Суть архитектуры пайплайнов: линейная цепочка независимых обработчиков, связанных только через статус батча в БД. Каждый пайплайн идемпотентен в части захвата и знает только свой участок работы. Повторяющийся код вынесен в общий модуль чистыми функциями — DRY без классов и наследования.
